# wgabrielong.github.io

Personal academic website for Wern Juin Gabriel Ong — built with
[Astro](https://astro.build/) + TypeScript, rendered to a static site for
GitHub Pages. No pre-built theme; everything here is hand-rolled.

This README is a working reference for editing the site. Read the relevant
section before making a change so edits stay consistent with the existing
structure and design.

---

## What this repo is (and what lives elsewhere)

This repo is the **website**: layout, styling, and the structured content
(publications, notes, talks, activities, bios) that the pages render.

**The CV is built here.** `cv-src/main.tex` is compiled by the deploy workflow
with Tectonic into `public/cv/main.pdf`, so the live site serves it at
`/cv/main.pdf` (see [Build & deploy status](#build--deploy-status)).

**Typeset notes live elsewhere.** Lecture and seminar notes are written and
compiled in a separate **`academic-writing`** repo, which is already built and
published at `wgabrielong.github.io/academic-writing/`. Notes entries here link
straight to those PDFs by **full URL** (e.g.
`https://wgabrielong.github.io/academic-writing/notes/.../Notes.pdf`) — this
repo neither hosts nor compiles them. Migrating older notes into
`academic-writing` is an ongoing, gradual process, so not everything is listed
yet; entries get added here as their PDFs go up.

### How content is loaded

`src/content.config.ts` defines four collections:

- `publications`, `notes`, `talks` — **glob loader**, one YAML file per entry
  (the file's slug becomes its `id`).
- `activities` — **file loader**, a single YAML file holding an array of terms
  (each with its own `id`).

Math in any title/heading is authored as inline LaTeX between `$…$` and
rendered with KaTeX via `renderInlineMath()` from `src/utils/math.ts`. KaTeX's
stylesheet is imported in `BaseLayout.astro`; `math.css` tunes it to match.

## Adding / editing content

### A publication — `src/content/publications/<prefix>-<nn>-<slug>.yaml`

**Filename drives grouping and order.** The prefix selects the section on the
Research page and the number sets order *within* it:

- `math-` → Mathematics, `soft-` → Software, `bio-` → Computational Biology.
- Numbers are **chronological (01 = oldest)**; the page displays each section
  **newest-first** (sorts by `id` descending). So the newest math paper gets
  the highest `math-NN`.

```yaml
title: '$\mathbb{A}^{1}$-Brouwer Degrees in Macaulay2'
titlePlain: '𝔸¹-Brouwer Degrees in Macaulay2'   # only when title has $…$ math
venue: 'Journal of Software for Algebra and Geometry, Vol. 14, No. 1'  # optional
status: published            # published | preprint | in prep
byline: 'with Nikita Borisov, Thomas Brazelton, and Andrew Tawfeek'
description: >-              # optional: short, always-visible blurb (software)
  A one- or two-sentence inline description.
abstract: >-                # optional: hidden behind an "Abstract" dropdown
  Long abstract text as a YAML folded block scalar. Write it as one flowing
  paragraph; the ">-" folds newlines into spaces and trims the trailing one.
  Inline math like $\mathbb{A}^{1}$ renders here too.
links:                       # optional; external URLs only (validated)
  - label: Journal
    url: 'https://doi.org/10.2140/jsag.2024.14.175'
  - label: arXiv
    url: 'https://doi.org/10.48550/arXiv.2312.00106'
pdf: 'https://wgabrielong.github.io/academic-writing/papers/.../main.pdf'  # optional
```

Field conventions:

- **`title`** — display title; may contain inline `$…$` math.
- **`titlePlain`** — *only* when `title` contains math. A markup-free version
  used for the HTML `title` tooltip / metadata / screen readers (use Unicode,
  e.g. `𝔸¹`). Plain-math-free titles don't need it.
- **`byline`** — defaults to `Wern Juin Gabriel Ong` (see the schema default).
  For co-authored work, override with the **`with A, B, and C`** convention
  (Gabriel implied as an author; list collaborators alphabetically by surname,
  matching how they're credited elsewhere).
- **`abstract`** vs **`description`** — `abstract` sits behind a `<details>`
  dropdown; `description` is a shorter blurb shown inline (used for the software
  entry). Both optional.
- **`links[].url`** must be an absolute URL (Zod `.url()`), unlike talks.
- **`pdf`** — optional full URL to the typeset PDF on the academic-writing site
  (e.g. `…/academic-writing/papers/<slug>/main.pdf`); renders as a "PDF" link
  after the arXiv/Journal links. (Notes use the same `pdf` field.)

### A note — `src/content/notes/<slug>.yaml`

One file per note set. The Notes page splits into two subsections — **Course
Notes** (`section: course`) and **Miscellany** (everything else) — and within
each sorts **newest term first** (term parsed as `Season YEAR`, with
`Winter YYYY/YY` counting as starting in `YYYY`; undated notes sort last).

```yaml
title: 'V5A2: Rigid Analytic Geometry'
section: course                    # course | miscellany (omit → Miscellany)
description: 'A sentence or two, shown inline.'  # optional (mainly miscellany)
courseCode: 'V5A2'                 # optional
instructor: 'Prof. Jens Franke'    # optional
institution: 'University of Bonn'  # optional
term: 'Summer 2025'                # optional
status: complete                   # optional; complete | incomplete
pdf: 'https://wgabrielong.github.io/academic-writing/notes/bonn-summer-25/V5A2-Rigid-Analytic-Geometry/Rigid_Analytic_Geometry_Notes.pdf'
```

- `pdf` is the **full URL** to the compiled PDF on the academic-writing site.
- Only `title` and `pdf` are required, so a standalone primer can be just those
  two plus `section: miscellany`.
- `status: incomplete` renders an "In progress" tag.

### A talk — `src/content/talks/talk-<nn>-<slug>.yaml`

One file per talk. The Talks page shows the **current + previous year** by
default and collapses older talks behind a "Show earlier talks" toggle — the
same `<details>`-based pattern as Activities. Recency is read from the `date`
field (academic-year spans like `Winter 2024/25` count as the ending year).

```yaml
title: '$\mathbb{A}^{1}$-Brouwer Degrees in Macaulay2'  # $…$ math OK
venue: 'Effective Methods in Algebraic Geometry — Leipzig'
date: 'July 2024'
type: talk                         # slides | poster | talk
links:                             # optional — omit entirely if none yet
  - label: Slides
    url: 'https://…/Slides.pdf'
```

- **`title`**, **`venue`**, **`date`**, **`type`** are required.
- **`type`** categorizes the format (posters get a "Poster" tag, otherwise
  "Talk").
- **`links`** is optional and may be omitted (or left `[]`) for a talk with no
  materials yet — better an empty list than a dead placeholder link. `url` is a
  plain string, so relative/future paths are allowed (unlike publications).
- Filenames are numbered (`talk-01`, …) only as a stable id / tiebreaker;
  display order is by year (newest first), not by filename.

### An activity — `src/content/activities/activities.yaml`

A single file: an array of term objects, each with an `id`, a `term` label, and
an `entries` list. The Activities page shows the **current year + previous full
year** by default (computed from the build date) and hides older terms behind a
"Show earlier activities" toggle.

```yaml
- id: fall-2026
  term: 'Fall 2026'
  entries:
    - dateRange: '9th–13th November'   # standardized: en-dash, day before month, no year
      title: 'Arithmetic Geometry and Higher Algebra'
      institution: 'University of Copenhagen'   # optional
      location: 'Copenhagen, Denmark'           # optional
```

- **`location` convention:** `City, ST, USA` (2-letter state) for the US,
  `City, Country` otherwise, `(Virtual)` for remote attendance.
- Terms sort newest-first automatically (`Season YEAR`); entries render in file
  order, so list them newest-first within a term.
- One term-group renders via `components/ActivityTerm.astro` (three type tiers:
  title > institution > location).

## Editing the bios

- **About-page bio** — `src/pages/index.astro`, the `<section class="bio">`
  paragraphs. The "About My Name" section is just below it in the same file.
- **CV longer bio + Fun Facts** — `src/components/cv-bio.md`. This is plain
  **Markdown** (headings + links), imported into `src/pages/cv/index.astro` and
  rendered above the CV download link. Edit the prose/links here; the page
  itself only holds the heading and the PDF link block (`pdfPath = /cv/main.pdf`
  — leave that as-is; it's produced by CI).

## Design system

Defined as CSS custom properties in `src/styles/global.css`. Reuse these tokens
rather than hardcoding values, so edits stay visually consistent.

| Token | Value | Use |
| --- | --- | --- |
| `--color-bg` | `#F7F8FA` | Page background (light neutral, not white) |
| `--color-border` | `#E2E5EA` | Hairline rules, dividers, pills |
| `--color-heading` | `#111111` | Headings |
| `--color-body` | `#2E333A` | Body text |
| `--color-muted` | `#5B6472` | Secondary text, nav, dates, metadata |
| `--color-accent` | `#00458E` | Uni Bonn "Congress Blue" — links, hover, role line |
| `--color-accent-hover` | `#003468` | Accent hover |

Rules of thumb:

- **Font:** Inter (weights 400 / 500 / 600), loaded in `BaseLayout.astro`.
- **The accent is used sparingly** — links, hover states, the role line, an
  occasional label. Don't fill regions with it.
- **Body:** `1.75` line-height. **Headings:** `-0.01em` letter-spacing, weight
  600. Content column caps at `--content-width` (`46rem`).
- **Math** styling lives in `math.css`; don't restyle KaTeX ad hoc.
- New pages should wrap content in `BaseLayout` and use the `.container` /
  existing type styles.

## Build & deploy status

**Live — the site auto-deploys to GitHub Pages from `main`.**

- `.github/workflows/build.yml` ("Build & Deploy") runs on every push to `main`
  (and manual dispatch): it installs **Tectonic** via `wtfjoke/setup-tectonic`,
  compiles `cv-src/main.tex` to `public/cv/main.pdf`, runs `npm ci` +
  `astro build`, then deploys `dist/` to Pages via
  `actions/upload-pages-artifact` + `actions/deploy-pages`.
- Pages **Source** is set to **GitHub Actions** and `main` is the default
  branch, so pushing to `main` rebuilds and redeploys `wgabrielong.github.io`.
- `npm run build` still produces the full static site in `dist/` for local
  checks.

Not yet wired: a **PR-validation CI check** (build + schema check on pull
requests) — see the note in the next section.

## Editing content via GitHub's web interface

Now that the site is live, **this is the intended primary way to make content
edits** (updating a publication, adding a talk, fixing a note's metadata).
Larger structural changes are done through Claude Code.

Because a push to `main` deploys immediately, prefer the PR flow below until the
build/schema PR check exists — a malformed frontmatter would otherwise go live.

1. Navigate to the file on GitHub — e.g. `src/content/talks/…yaml`.
2. Click the **pencil (✏️) icon** to edit in the browser.
3. Make the change following the frontmatter formats above.
4. When committing, **always choose "Create a new branch for this commit and
   start a pull request"** — do **not** commit directly to the branch that
   deploys.

Why the PR: once a CI check that **validates PRs is in place** (schema + build
check — *not yet built*), opening a PR runs the build against your change before
anything can go live, so a malformed YAML frontmatter or broken link gets caught
instead of taking down the deploy. Until that check exists, the PR habit is
still the right default to adopt.
